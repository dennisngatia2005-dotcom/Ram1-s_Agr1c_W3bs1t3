Blockchain Tables

Oracle 21c NEW

Cryptographically tamper-evident, append-only ledger tables

What is it?

A Blockchain Table is a special Oracle table where each row is cryptographically chained to the row inserted before it using a hash algorithm (SHA2_512 in our case). Once a row is inserted, it cannot be updated or deleted — the database enforces this at the engine level, not the application level.

The "chain" means that if anyone attempts to tamper with the data outside of Oracle (via OS-level file editing or backup restoration), the chain verification will fail and Oracle will report the corruption.

Why we use it: Every payment made by a member is automatically written to paymentAuditLedger by a trigger. This gives us a forensic-grade payment history that no admin, developer or DBA can erase or alter — regardless of their database privileges.
How it works in AgriSoc: The trg_payment_blockchain trigger fires AFTER every INSERT on payment and writes a row to the blockchain table. The NO DELETE UNTIL 31 DAYS AFTER INSERT clause means rows survive at least 31 days. NO DROP UNTIL 31 DAYS IDLE prevents the table from being dropped while recent data exists.
-- Oracle 21c: Create a Blockchain Table CREATE BLOCKCHAIN TABLE "paymentAuditLedger" ( "ledgerID" NUMBER GENERATED BY DEFAULT AS IDENTITY, "paymentID" NUMBER NOT NULL, "memberID" NUMBER NOT NULL, "amount" NUMBER(10,2) NOT NULL, "paymentType" VARCHAR2(30) NOT NULL, "actionTime" TIMESTAMP DEFAULT SYSTIMESTAMP, "actionBy" VARCHAR2(60) DEFAULT USER ) NO DROP UNTIL 31 DAYS IDLE NO DELETE UNTIL 31 DAYS AFTER INSERT HASHING USING "SHA2_512" VERSION "v1"; -- Trigger that feeds it automatically CREATE OR REPLACE TRIGGER "trg_payment_blockchain" AFTER INSERT ON "payment" FOR EACH ROW BEGIN INSERT INTO "paymentAuditLedger" ("paymentID", "memberID", "amount", "paymentType") VALUES (:NEW."paymentID", :NEW."memberID", :NEW."amount", :NEW."paymentType"); END; /

Immutable Tables

Oracle 21c NEW

Append-only tables for historical archives — lighter than blockchain, without cryptographic chaining

Immutable Tables are similar to Blockchain Tables in that rows cannot be deleted or updated, but they do not chain rows cryptographically. They are a lighter-weight option for archiving data where you want immutability enforced by the database but don't need tamper-evidence verification.

Why we use it: borrowHistory is an immutable table that receives a copy of every completed borrow (when returnDate is set). This gives us a permanent, unalterable library history — useful for annual reporting and member dispute resolution.
How it works: The trg_borrow_archive trigger fires AFTER UPDATE on returnDate (when a resource is returned) and inserts a snapshot row into borrowHistory. The NO DELETE UNTIL 90 DAYS AFTER INSERT policy keeps records for a minimum of 90 days.
-- Oracle 21c: Immutable Table (no crypto chaining) CREATE IMMUTABLE TABLE "borrowHistory" ( "historyID" NUMBER GENERATED BY DEFAULT AS IDENTITY, "borrowID" NUMBER NOT NULL, "memberName" VARCHAR2(62) NOT NULL, "resourceTitle" VARCHAR2(255)NOT NULL, "borrowDate" DATE NOT NULL, "returnDate" DATE, "archivedAt" TIMESTAMP DEFAULT SYSTIMESTAMP ) NO DROP UNTIL 90 DAYS IDLE NO DELETE UNTIL 90 DAYS AFTER INSERT; -- Blockchain vs Immutable — key difference: -- Blockchain: rows chained + hash verified = tamper-EVIDENT -- Immutable: rows protected but no chain = tamper-RESISTANT

Native JSON Type Column

Oracle 21c NEW

JSON stored in Oracle's optimised binary OSON format — not CLOB or VARCHAR2

Before Oracle 21c, JSON was stored in VARCHAR2 or CLOB columns and Oracle would parse the text on every query. In 21c, the JSON data type stores documents in OSON (Oracle Optimised Storage for JSON) — a binary format that is faster to read, smaller on disk, and natively understood by the query engine.

You can query JSON fields using JSON_VALUE(), JSON_QUERY(), dot-notation, or function-based indexes on specific paths.

Why we use it: Announcements have a metadata JSON column storing flexible per-announcement data: priority, author, tags. This avoids creating separate columns for every possible attribute — the schema stays clean while the data stays rich.
How we query it: JSON_VALUE("metadata", '$.priority') extracts the priority field inline in SQL. We also have a function-based index on JSON_VALUE("metadata", '$.priority') so filtering by priority uses the index rather than scanning OSON blobs.
-- 21c: Native JSON column (OSON binary, not CLOB) ALTER TABLE "announcement" ADD ("metadata" JSON); -- Insert with JSON_OBJECT constructor INSERT INTO "announcement" ("title", "body", "metadata") VALUES ( 'Drought Advisory', 'Farmers are advised...', JSON_OBJECT( 'priority' VALUE 'high', 'author' VALUE 'Admin', 'tags' VALUE 'drought,crops' ) ); -- Query using JSON_VALUE SELECT "title", JSON_VALUE("metadata", '$.priority') AS priority, JSON_VALUE("metadata", '$.author') AS author FROM "announcement" WHERE JSON_VALUE("metadata", '$.priority') = 'high'; -- Function-based index speeds up the above query CREATE INDEX "idx_announcement_priority" ON "announcement" (JSON_VALUE("metadata", '$.priority'));

SQL Macros

Oracle 21c NEW

Reusable SQL fragments inlined at parse time — zero runtime overhead

SQL Macros are PL/SQL functions that return a SQL fragment as a VARCHAR2 string. The key distinction: the returned SQL is inlined into the calling query at parse time by the Oracle optimizer — the function body never runs at query execution time.

There are two types: Scalar Macros (return a single value expression, usable in SELECT/WHERE) and Table Macros (return a full SELECT statement, usable in the FROM clause like a parameterised view).

Why vs. pipelined functions or views? A pipelined function runs at execution time and returns rows via PGA memory — slower. A view cannot accept parameters. A SQL Macro gives you the parameterisation of a function with the performance of a view — the optimizer can see through it and use indexes.
Our macros: (1) member_tenure_years(joinDate) — scalar macro computing membership duration, shown on every member profile. (2) active_borrows_for(memberID) — table macro for a member's current borrows. (3) revenue_in_period(from, to) — table macro for revenue analytics over a custom date range.
-- SCALAR SQL Macro: inlines as an expression CREATE OR REPLACE FUNCTION member_tenure_years( p_joinDate DATE ) RETURN VARCHAR2 SQL_MACRO(SCALAR) IS BEGIN RETURN q'[ ROUND(MONTHS_BETWEEN(SYSDATE, p_joinDate) / 12, 1) ]'; END; / -- Used in a SELECT — inlined, not called SELECT "fullName", member_tenure_years("joinDate") AS tenure_years FROM "member"; -- TABLE SQL Macro: parameterised virtual table CREATE OR REPLACE FUNCTION active_borrows_for( p_memberID NUMBER ) RETURN VARCHAR2 SQL_MACRO(TABLE) IS BEGIN RETURN q'[ SELECT b."borrowID", r."title", b."borrowDate" FROM "borrow" b JOIN "resource" r ON b."resourceID" = r."resourceID" WHERE b."memberID" = p_memberID AND b."returnDate" IS NULL ]'; END; / -- Called like a table — optimizer inlines it SELECT * FROM active_borrows_for(101);

Virtual Columns

Computed columns — always current, never stored on disk

A Virtual Column is defined as an expression involving other columns in the same row. Oracle evaluates the expression on the fly whenever the column is queried. The value is never written to disk — there is zero storage overhead.

Virtual columns can be indexed, used in WHERE clauses, and appear in views. They are especially useful for computed business fields that must always reflect the current base data.

Our use: fullName on the member table is a virtual column: TRIM("firstName") || ' ' || TRIM("secondName"). Every query, procedure and view that needs the full name uses this column — there is no risk of firstName and secondName being updated while fullName stays stale.
-- Add a virtual column to existing table ALTER TABLE "member" ADD ( "fullName" VARCHAR2(62) GENERATED ALWAYS AS (TRIM("firstName") || ' ' || TRIM("secondName")) VIRTUAL ); -- It works like any column but is never stored: SELECT "memberID", "fullName", "joinDate" FROM "member"; -- fullName is computed fresh each time -- Can be indexed for fast full-name lookups: CREATE INDEX "idx_member_fullName" ON "member" (UPPER("fullName"));

Invisible Columns

Hidden from SELECT * but fully accessible and indexed by name

An Invisible Column is present in the table and fully functional, but it does not appear in SELECT *, %ROWTYPE declarations, or DESCRIBE output. It must be referenced explicitly by name.

This is ideal for sensitive fields — like password hashes — that should never accidentally be exposed by a generic query. It also allows columns to be added to a table without breaking existing code that uses SELECT *.

Our use: "pwdHash" on the member table stores the bcrypt-hashed password. Being invisible means it is never accidentally returned by SELECT * FROM "member" — it must be explicitly requested: SELECT "pwdHash" FROM "member".
-- Add an invisible column ALTER TABLE "member" ADD ( "pwdHash" VARCHAR2(64) INVISIBLE ); -- SELECT * will NOT return "pwdHash": SELECT * FROM "member"; -- Result: memberID, firstName, secondName, email... (no pwdHash) -- Must be referenced explicitly: SELECT "memberID", "pwdHash" FROM "member" WHERE "email" = 'user@example.com'; -- Make visible again at any time: -- ALTER TABLE "member" MODIFY "pwdHash" VISIBLE;

Interval Range Partitioning

Automatic yearly partition creation — older data stays queryable but physically isolated

Partitioning splits a large table into smaller physical segments (partitions) while presenting a single logical table to queries. With INTERVAL partitioning, Oracle automatically creates new partitions as data arrives — you define only the seed partitions and the interval rule.

Our use: paymentArchive is range-partitioned by paymentDate with a NUMTOYMINTERVAL(1,'YEAR') interval. A query like WHERE paymentDate BETWEEN '2024-01-01' AND '2024-12-31' hits only the 2024 partition — partition pruning eliminates all other segments from the scan.
Performance benefit: Year-based revenue reports (which are common in the admin dashboard) execute in milliseconds even as the archive grows to millions of rows, because the optimizer restricts the scan to a single physical partition.
-- INTERVAL RANGE partition by year -- New year partitions auto-create on first insert CREATE TABLE "paymentArchive" ( "archiveID" NUMBER GENERATED BY DEFAULT AS IDENTITY, "memberID" NUMBER NOT NULL, "amount" NUMBER(10,2) NOT NULL, "paymentDate" DATE NOT NULL ) PARTITION BY RANGE ("paymentDate") INTERVAL (NUMTOYMINTERVAL(1, 'YEAR')) ( PARTITION pmt_2023 VALUES LESS THAN (DATE '2024-01-01'), PARTITION pmt_2024 VALUES LESS THAN (DATE '2025-01-01') -- 2025, 2026... created automatically ); -- Query uses partition pruning automatically: SELECT SUM("amount") FROM "paymentArchive" WHERE "paymentDate" BETWEEN DATE '2024-01-01' AND DATE '2024-12-31'; -- Only the pmt_2024 segment is scanned

Property Graph

Oracle 21c

Native graph model over relational data — members, resources and branches as vertices and edges

Oracle Property Graphs define a graph schema over existing relational tables. Vertex tables become nodes and edge tables become relationships. The graph is not a copy of the data — it is a live graph view over the same rows.

Queries use GRAPH_TABLE() with SQL/PGQ syntax (SQL Property Graph Query) to traverse relationships, find paths, and calculate graph metrics — all within a standard SQL SELECT.

Our graph: Members and Resources are vertices. Borrow events are edges (Borrowed), and the Member→Branch relationship is also an edge (BelongsTo). This lets us answer questions like: "Which members share the same borrowing pattern?" and "Which branch has the most connected members?"
Visualisation: The D3.js network graph on the Admin Dashboard fetches member and resource nodes from the API (/api/graph-data) and borrow edges, then renders them as an interactive force-directed graph. Drag nodes to explore the borrow network.
-- Define the Property Graph CREATE OR REPLACE PROPERTY GRAPH "agriGraph" VERTEX TABLES ( "member" KEY ("memberID") LABEL Member PROPERTIES ("firstName", "joinDate"), "resource" KEY ("resourceID") LABEL Resource PROPERTIES ("title", "type") ) EDGE TABLES ( "borrow" KEY ("borrowID") SOURCE KEY ("memberID") REFERENCES "member" DESTINATION KEY ("resourceID") REFERENCES "resource" LABEL Borrowed ); -- Query: members who borrowed resources (GRAPH_TABLE) SELECT member_id, first_name, borrow_count FROM GRAPH_TABLE ( "agriGraph" MATCH (m IS Member) -[b IS Borrowed]-> (r IS Resource) COLUMNS ( m."memberID" AS member_id, m."firstName" AS first_name, COUNT(*) AS borrow_count ) ) ORDER BY borrow_count DESC;

Index Strategy

B-tree, composite, function-based, and invisible indexes tuned to our exact query patterns

B-tree Indexes (standard)

Applied to every foreign key column (branchID, categoryID, memberID) and the eventDate column on shows. These prevent full table scans on JOIN operations and range queries.

Composite Index

idx_borrow_active on ("memberID", "returnDate") matches the most common borrow query pattern: WHERE memberID = :x AND returnDate IS NULL. The leading column is the high-cardinality filter, the trailing column enables the IS NULL check without a table access.

Function-Based Indexes

idx_member_email_upper on UPPER("email") supports case-insensitive email lookups in login and search. idx_member_fullName on UPPER(fullName) supports the member search procedure.

Invisible Index

idx_payment_type_invis on paymentType is INVISIBLE — maintained by DML but ignored by the optimizer unless hinted. This lets us test whether the index would help a query before committing to making it visible, with zero risk to existing query plans.

-- Standard B-tree: foreign key CREATE INDEX "idx_member_branchID" ON "member" ("branchID"); -- Composite: matches (memberID, returnDate IS NULL) CREATE INDEX "idx_borrow_active" ON "borrow" ("memberID", "returnDate"); -- Function-based: case-insensitive email login CREATE INDEX "idx_member_email_upper" ON "member" (UPPER("email")); -- Function-based: full-name search CREATE INDEX "idx_member_fullName" ON "member" (UPPER("fullName")); -- JSON field index: filter announcements by priority CREATE INDEX "idx_announcement_priority" ON "announcement" (JSON_VALUE("metadata", '$.priority')); -- INVISIBLE: test before committing CREATE INDEX "idx_payment_type_invis" ON "payment" ("paymentType") INVISIBLE; -- Use invisible index with hint: SELECT /*+ INDEX("payment" "idx_payment_type_invis") */ * FROM "payment" WHERE "paymentType" = 'Annual Fee';

Roles & Least-Privilege Access

Three layered roles — readonly → member → admin — each with precisely scoped grants

Oracle Roles group related privileges and can be granted to users or other roles (role inheritance). This allows us to build a privilege hierarchy where each level inherits from the one below while adding its own capabilities.

Role inheritance: each arrow means "inherits all privileges from"

-- Create the three roles CREATE ROLE agri_readonly; CREATE ROLE agri_member_role; CREATE ROLE agri_admin_role; -- agri_readonly: public-facing SELECT only GRANT SELECT ON "branch" TO agri_readonly; GRANT SELECT ON "resource" TO agri_readonly; GRANT SELECT ON "show" TO agri_readonly; -- agri_member_role inherits readonly + adds DML GRANT agri_readonly TO agri_member_role; GRANT INSERT ON "borrow" TO agri_member_role; GRANT INSERT ON "payment" TO agri_member_role; GRANT UPDATE ("returnDate") ON "borrow" TO agri_member_role; GRANT EXECUTE ON "borrowResource" TO agri_member_role; GRANT EXECUTE ON "makePayment" TO agri_member_role; -- agri_admin_role inherits member + full control GRANT agri_member_role TO agri_admin_role; GRANT INSERT, UPDATE, DELETE ON "member" TO agri_admin_role; GRANT SELECT ON "memberAuditLog" TO agri_admin_role; GRANT SELECT ON "paymentAuditLedger" TO agri_admin_role; -- Assign to database users: -- GRANT agri_admin_role TO agri_web_admin;
