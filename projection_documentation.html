What is AgriSoc Kenya?

The Agricultural Society of Kenya is a (convincingly real) membership organisation serving crop farmers, livestock keepers, agri-businesses and students across four branches: Nairobi Central, Rift Valley, Coast and Western Kenya. Members can join, borrow library resources, make payments, attend shows, and read announcements. Admins get a full analytics dashboard, an audit trail, and enough charts to make a data analyst weep with joy.

The actual goal is to demonstrate deep, production-quality Oracle 21c knowledge. The farming society is the stage. Oracle 21c is the show.

Why this structure?

Flask's Application Factory pattern (create_app() in app.py) is the recommended approach for anything bigger than a single-file script. It separates configuration from code and makes the app easier to test. Blueprints split routes into logical groups — changes to admin logic never touch member logic.

The db/ module is deliberately separate from routes. The Oracle connection pool is created once at startup and shared across all requests. Oracle connections are expensive to open — using a pool means we open 2–10 connections total, not one per HTTP request.

The .env file contains the database password. It is explicitly listed in .gitignore. We mention this because someone, somewhere has committed a production database password to a public GitHub repo, watched their AWS bill hit $4,000 in six hours, and written a very humbling Medium article about it. We are not that person. We have read that article twice.

Section 05 — Oracle 21c Feature

Blockchain Tables

Tamper-evident, SHA2_512-chained payment history that not even SYSDBA can erase

What is a Blockchain Table?

A Blockchain Table looks and feels like a regular Oracle table from the outside. Under the hood, every row that is inserted gets cryptographically linked to the row before it using SHA2_512. The hash chain is maintained by the Oracle engine itself — not your application code, not a trigger, not a cron job. Oracle. The database. Just does it. Automatically.

The consequences are profound:

  • Rows cannot be updated — full stop, no exceptions
  • Rows cannot be deleted until the retention period expires
  • The table cannot be dropped while it contains recent data
  • Even a user with SYSDBA cannot silently alter the chain — Oracle will detect it
  • You can call DBMS_BLOCKCHAIN_TABLE.VERIFY_ROWS at any time to confirm integrity
Why we use it: Every time a member makes a payment, the trg_payment_blockchain trigger automatically writes a copy to paymentAuditLedger. This gives the society an unchallengeable financial audit trail. If a member disputes a charge or a treasurer is accused of fraud, the blockchain table provides irrefutable cryptographic evidence. Signed by Oracle's own hash engine. No application code required.
Before blockchain tables existed, "immutable audit trail" meant "we pinky-promise we haven't touched the audit table." Oracle 21c made pinky-promises legally inadmissible. Your DBA will find this either deeply reassuring or personally offensive, depending on how much they trust themselves. Both reactions are valid.
Flask connection: routes/members.py → make_payment() → makePayment stored proc → INSERT into payment → trigger fires → row inserted into blockchain table. The Flask application never directly touches the blockchain table. Zero application code needed for the audit.
oracle_advanced_21c.sql — Blockchain Table 
-- Oracle 21c: CREATE BLOCKCHAIN TABLE
-- This is NOT a regular table. Normal rules don't apply.
CREATE BLOCKCHAIN TABLE "paymentAuditLedger" (
    "ledgerID"    NUMBER
        GENERATED BY DEFAULT AS IDENTITY,
    "paymentID"   NUMBER       NOT NULL,
    "memberID"    NUMBER       NOT NULL,
    "amount"      NUMBER(10,2) NOT NULL,
    "paymentType" VARCHAR2(30) NOT NULL,
    "actionTime"  TIMESTAMP    DEFAULT SYSTIMESTAMP,
    "actionBy"    VARCHAR2(60) DEFAULT USER
)
NO DROP UNTIL 31 DAYS IDLE
NO DELETE UNTIL 31 DAYS AFTER INSERT
HASHING USING "SHA2_512" VERSION "v1";
-- ↑ Every INSERT is hashed and chained to the
--   previous row. Automatically. By the engine.

-- The trigger that populates it
CREATE OR REPLACE TRIGGER "trg_payment_blockchain"
AFTER INSERT ON "payment"
FOR EACH ROW
BEGIN
    INSERT INTO "paymentAuditLedger"
        ("paymentID", "memberID",
         "amount",    "paymentType")
    VALUES
        (:NEW."paymentID", :NEW."memberID",
         :NEW."amount",    :NEW."paymentType");
END;
/

-- Verify chain integrity at any time
BEGIN
    DBMS_BLOCKCHAIN_TABLE.VERIFY_ROWS(
        schema_name => 'AGRISOC',
        table_name  => 'paymentAuditLedger'
    );
END;
/
-- Returns: 0 rows tampered. Every time. Or it raises
-- an exception and someone has a very bad day.

Immutable Tables

Append-only archive storage — lighter than blockchain, still impossible to quietly alter

Blockchain vs Immutable — The Distinction

PropertyBlockchain TableImmutable Table
Append-only (no UPDATE)✅ Yes✅ Yes
No DELETE until retention✅ Yes✅ Yes
Hash chaining (SHA2_512)✅ Yes❌ No
Cryptographic verification✅ VERIFY_ROWS❌ No API
Performance overheadHigher (hashing)Lower
Our use casePayment ledgerBorrow archive
Why we use it: borrowHistory stores every completed borrow as a permanent, append-only record. We deliberately denormalise — storing the member's full name and resource title directly — so the history remains readable even if the source member or resource is later deleted. Perfect for annual library reports and proving that yes, James did borrow the Seed Catalogue of 1987 and never returned it.
We used an immutable table rather than a blockchain table for borrow history because borrow records don't need forensic tamper-evidence — they just need to exist forever. Think of it as the difference between a notarised affidavit and a laminated sticky note. Both permanent. One slightly more court-admissible. We chose the laminated sticky note. It was faster to implement.
Immutable Table + Archive Trigger 
-- Oracle 21c: CREATE IMMUTABLE TABLE
CREATE IMMUTABLE TABLE "borrowHistory" (
    "historyID"      NUMBER
        GENERATED BY DEFAULT AS IDENTITY,
    "borrowID"       NUMBER        NOT NULL,
    "memberID"       NUMBER        NOT NULL,
    "resourceID"     NUMBER        NOT NULL,
    "memberName"     VARCHAR2(62)  NOT NULL, -- denorm
    "resourceTitle"  VARCHAR2(255) NOT NULL, -- denorm
    "borrowDate"     DATE          NOT NULL,
    "returnDate"     DATE,
    "archivedAt"     TIMESTAMP DEFAULT SYSTIMESTAMP
)
NO DROP UNTIL   90 DAYS IDLE
NO DELETE UNTIL 90 DAYS AFTER INSERT;

-- Trigger: fires when returnDate is set for first time
CREATE OR REPLACE TRIGGER "trg_borrow_archive"
AFTER UPDATE OF "returnDate" ON "borrow"
FOR EACH ROW
WHEN (NEW."returnDate" IS NOT NULL AND
      OLD."returnDate" IS NULL)
DECLARE
    v_name  VARCHAR2(62);
    v_title VARCHAR2(255);
BEGIN
    -- Grab denormalised values for permanent record
    SELECT "fullName" INTO v_name
    FROM   "member"
    WHERE  "memberID" = :NEW."memberID";

    SELECT "title" INTO v_title
    FROM   "resource"
    WHERE  "resourceID" = :NEW."resourceID";

    INSERT INTO "borrowHistory"
        ("borrowID",    "memberID",
         "resourceID",  "memberName",
         "resourceTitle","borrowDate", "returnDate")
    VALUES
        (:NEW."borrowID",   :NEW."memberID",
         :NEW."resourceID", v_name, v_title,
         :NEW."borrowDate", :NEW."returnDate");
END;
/
Section 07 — Oracle 21c Feature

Native JSON Data Type

OSON binary storage — not VARCHAR2, not CLOB. A first-class JSON type the engine understands natively.

Why Not Just Use VARCHAR2?

Before Oracle 21c, developers stored JSON in VARCHAR2 or CLOB columns with a CHECK (col IS JSON) constraint. Oracle would parse the text string on every single access — turning a structured document into a parsing exercise every time you ran a query. Fine for small documents and low traffic. Noticeably painful otherwise.

Oracle 21c introduced a dedicated JSON type that stores documents in OSON (Oracle Optimised Storage for JSON) — a binary format that is faster to parse, supports lazy field access, and is deeply understood by the query engine. No text conversion. No full-document deserialisation to read one field.

Our use: The announcement.metadata column is typed as JSON. Every announcement stores priority, author name, and tags as a JSON object. Querying JSON_VALUE("metadata", '$.priority') reads only that field from the binary OSON — and our function-based index on that path makes high-priority lookups instant.
We briefly considered storing metadata as separate columns: priority, author, tags. Then we realised we'd want to add attachment_url next week, expiry_date the week after, and target_branch the week after that. Within a month the table would have 40 nullable columns for edge cases. One JSON column: zero column sprawl. Oracle: "I told you so." We listened. The table has five columns and we have our dignity.
Native JSON Column + Queries 
-- Oracle 21c: native JSON column (binary OSON)
-- Not VARCHAR2. Not CLOB. Actual JSON type.
"metadata" JSON

-- Insert using JSON_OBJECT constructor
INSERT INTO "announcement"
    ("title", "body", "metadata")
VALUES (
    'Drought Advisory',
    'Please conserve water and stagger irrigation...',
    JSON_OBJECT(
        'priority' VALUE 'high',
        'author'   VALUE 'Branch Secretary',
        'tags'     VALUE 'drought,water,crops'
    )
);

-- Read with JSON_VALUE — reads only that field
-- from binary OSON. No full document parse.
SELECT
    "title",
    JSON_VALUE("metadata", '$.priority') AS priority,
    JSON_VALUE("metadata", '$.author')   AS author,
    JSON_VALUE("metadata", '$.tags')     AS tags
FROM   "announcement"
WHERE  JSON_VALUE("metadata",'$.priority') = 'high';
-- ↑ hits idx_announcement_priority FBI, not full scan

-- In the announcementSummary view:
CREATE OR REPLACE VIEW "announcementSummary" AS
SELECT
    "title", "body", "postedDate",
    JSON_VALUE("metadata",'$.priority') AS priority,
    JSON_VALUE("metadata",'$.author')   AS author,
    JSON_VALUE("metadata",'$.tags')     AS tags
FROM  "announcement"
WHERE "isActive" = 1;
Section 08 — Oracle 21c Feature

SQL Macros

Parameterised SQL that is inlined by the optimizer at parse time. Zero runtime overhead.

How SQL Macros Work

A SQL Macro is a PL/SQL function that returns SQL text. But unlike a normal function, the returned text is substituted into the calling query before the optimizer builds a plan. The function body never executes at runtime — it only executes at parse time to produce the SQL fragment that gets inlined.

The result: the optimizer sees the macro's SQL as if you'd typed it directly. It can push predicates into it, use indexes, and compute optimal join orders — exactly like a hard-coded subquery. All the convenience of a function. None of the per-row execution overhead.

Two types:

  • Scalar Macro — inline expression, usable anywhere a column expression fits
  • Table Macro — inline SELECT, used in FROM clause like a parameterised view
Our macros:

member_tenure_years(joinDate) — Scalar. Returns years of membership as a decimal. Used on member profiles and in getMemberActivityReport.

active_borrows_for(memberID) — Table. Returns a member's active borrows. Called like a real table: SELECT * FROM active_borrows_for(101). Fully index-aware.

revenue_in_period(fromDate, toDate) — Table. Revenue breakdown by category for a date range. Powers the admin custom analytics query tool.
SQL Macros are what happens when Oracle engineers ask: "What if views could take parameters?" and then spend several years making it work properly without sacrificing optimizer intelligence. The result is a function that completely disappears at execution time, leaving only clean, optimized SQL in its place. Our PL/SQL developers called it cheating. We called it Oracle 21c doing exactly what we paid for. Everyone agreed the membership tenure calculation was cleaner for it.
SQL Macros — Scalar and Table 
-- SCALAR MACRO: used as an expression in SELECT/WHERE
-- Returns SQL text, not a value. Inlined at parse time.
CREATE OR REPLACE FUNCTION member_tenure_years(
    p_joinDate DATE
) RETURN VARCHAR2 SQL_MACRO(SCALAR)
IS BEGIN
    RETURN q'[
        ROUND(MONTHS_BETWEEN(SYSDATE, p_joinDate) / 12, 1)
    ]';
END;
/

-- Usage — function body does NOT run at query time
SELECT
    "fullName",
    member_tenure_years("joinDate") AS tenure_years
FROM "member"
WHERE member_tenure_years("joinDate") > 5;
-- Oracle sees: WHERE ROUND(MONTHS_BETWEEN(...)/12,1) > 5
-- Not: call Python function for every row. World of difference.

-- TABLE MACRO: a parameterised view
CREATE OR REPLACE FUNCTION active_borrows_for(
    p_memberID NUMBER
) RETURN VARCHAR2 SQL_MACRO(TABLE)
IS BEGIN
    RETURN q'[
        SELECT
            b."borrowID",
            r."title"      AS resource_title,
            b."borrowDate",
            SYSDATE - b."borrowDate" AS days_out
        FROM   "borrow"   b
        JOIN   "resource" r
            ON b."resourceID" = r."resourceID"
        WHERE  b."memberID"   = p_memberID
          AND  b."returnDate" IS NULL
    ]';
END;
/

-- Called like a real table — optimizer sees the SQL
SELECT * FROM active_borrows_for(101);

-- TABLE MACRO: custom date-range revenue breakdown
CREATE OR REPLACE FUNCTION revenue_in_period(
    p_from DATE, p_to DATE
) RETURN VARCHAR2 SQL_MACRO(TABLE)
IS BEGIN
    RETURN q'[
        SELECT
            c."categoryName",
            SUM(p."amount") AS total_revenue,
            COUNT(*)        AS payment_count
        FROM   "payment"  p
        JOIN   "member"   m ON p."memberID"  = m."memberID"
        JOIN   "category" c ON m."categoryID"= c."categoryID"
        WHERE  p."paymentDate" BETWEEN p_from AND p_to
        GROUP BY c."categoryName"
    ]';
END;
/
Section 09 — Oracle 21c Feature

Virtual & Invisible Columns

Computed columns that never go stale, and sensitive columns that hide from SELECT *

Virtual Column: fullName

A virtual column is defined as a SQL expression. Oracle evaluates it on-the-fly during queries. Nothing is stored on disk — zero storage overhead. The value is always exactly consistent with the base columns, automatically, forever, with no application effort.

Virtual Column Definition 
ALTER TABLE "member" ADD (
    "fullName" VARCHAR2(62)
    GENERATED ALWAYS AS
    (TRIM("firstName") || ' ' || TRIM("secondName"))
    VIRTUAL
);
-- Used in every stored procedure and view
-- that needs a displayable name.
-- No UPDATE ever needed. No stale data possible.
Before virtual columns there were exactly two options: store fullName (and risk it becoming stale when firstName is updated, which will happen, it always happens, someone always updates a first name without touching fullName), or concatenate in every query (and risk a developer forgetting the TRIM, producing "James Mwangi" with a rogue double space that only manifests in PDFs and printed reports). Virtual columns eliminated both failure modes. One expression, defined once, correct always.

Invisible Column: pwdHash

An invisible column exists and functions exactly like a normal column — it stores data, can be indexed, and can be queried — but it does not appear in SELECT *, %ROWTYPE, or DESCRIBE. You must reference it by name explicitly. This is precisely what you want for security-sensitive fields.

Invisible Column + Login Query 
ALTER TABLE "member" ADD (
    "pwdHash" VARCHAR2(64) INVISIBLE
);

-- SELECT * will NEVER return pwdHash
SELECT * FROM "member";
-- ↑ Safe to pass results anywhere — no hash leakage

-- Must be named explicitly to retrieve it
SELECT
    "memberID",
    "fullName",
    "pwdHash"   -- explicit — intentional access only
FROM  "member"
WHERE UPPER("email") = UPPER(:email);
-- ↑ Used only in the login verification flow

-- In Flask (members.py login handler):
-- row = fetchone(sql, {'email': email})
-- if bcrypt.checkpw(password, row['pwdHash']):
--     session['member_id'] = row['memberID']
Flask connection: members.login() queries "pwdHash" explicitly, then calls bcrypt.checkpw(). If the hash is NULL (first login or reset), it falls back to accepting the memberID as a one-time PIN, then prompts for a new password.
Section 10

Index Strategy

Nine indexes — B-tree, composite, function-based, JSON path, and invisible — each solving a specific query problem

Indexes are like the index at the back of a textbook. Without them, Oracle reads every single row in the table to find what you're looking for — this is called a "full table scan" and is Oracle's polite way of saying "you have disappointed me and I am reading the entire book to find page 47." With the right indexes, Oracle jumps straight to the answer. The art is choosing which columns to index, in what combination, and whether you need a standard index or something more specialised. We went fairly specialised.
Index NameTypeColumn(s)Problem It Solves
idx_member_branchID B-tree member.branchID Every member query JOINs with branch. Without this, every JOIN causes Oracle to scan the entire member table for each branch row. With it: instant lookup.
idx_borrow_memberID B-tree borrow.memberID Finding all borrows for a specific member — used on the dashboard, overdue report, and member activity report. Called constantly.
idx_borrow_active Composite (memberID, returnDate) The single most common borrow query: WHERE memberID = :x AND returnDate IS NULL. Both filter columns in one index scan. Without this, two separate index lookups or a full scan.
idx_member_email_upper Function-based UPPER("email") Case-insensitive email login. Without this FBI, WHERE UPPER(email) = :e requires a full table scan on every login attempt. With it: single row lookup regardless of email case.
idx_member_fullName Function-based UPPER("fullName") Admin member search by name. searchMembers procedure uses LIKE UPPER('%:q%') — index on the virtual column's uppercased value makes this fast even with wildcards.
idx_show_upcoming B-tree (eventDate, branchID) Homepage and shows page: WHERE eventDate >= SYSDATE ORDER BY eventDate. Composite index supports the range scan and eliminates a separate sort step.
idx_payment_memberID B-tree payment.memberID Member payment history page and the getMonthlyRevenue aggregation. Without this, every payment history query is a full table scan of potentially thousands of rows.
idx_announcement_priority JSON FBI JSON_VALUE("metadata",'$.priority') Filtering announcements by priority (high / normal / low) from the native JSON column — without deserialising the full OSON document for every row in the table.
idx_payment_type_invis Invisible payment.paymentType An invisible index used to test whether a paymentType index improves revenue-by-type queries. Invisible to the optimizer by default — enable with a hint to compare plans without committing.
Invisible index note: An invisible index is fully maintained by Oracle on every DML operation but is ignored by the optimizer unless you force it with /*+ INDEX(table idx_name) */. It is the correct way to test whether a new index would help before committing to it in production — you get real index maintenance cost without any query plan disruption.
Section 11

Triggers

Four database-level triggers that automate audit, archiving, blockchain feeds, and stock protection

A trigger is PL/SQL code that Oracle runs automatically when a table event occurs — completely outside the application. This is crucial for audit and security features because it means no application code path can bypass them. Not a bug. Not a test script. Not a rogue admin connecting directly via SQL*Plus at midnight. The trigger fires regardless.

⚡
trg_member_audit
Audit Trail

AFTER INSERT OR UPDATE OR DELETE ON "member"

Captures every change to the member table into memberAuditLog. Records the action type (INSERT / UPDATE / DELETE), the database user who made the change, the timestamp, and the old and new values of key fields.

Even if an admin bypasses the Flask application entirely and runs SQL directly in SQLDeveloper, this trigger still fires. It is impossible to change a member record without leaving a trace.

Flask path: admin.toggle_member() → UPDATE member → trigger → memberAuditLog row
⚡
trg_payment_blockchain
Blockchain Feed

AFTER INSERT ON "payment"

Every new payment row automatically feeds the paymentAuditLedger blockchain table. The Flask application calls makePayment(), which INSERTs into payment, which fires this trigger. The blockchain write is automatic — not configurable, not skippable.

This is the heart of the financial audit system. Zero lines of Flask code are responsible for blockchain integrity. Oracle handles it entirely.

Flask path: members.make_payment() → makePayment proc → INSERT payment → trigger → blockchain table
⚡
trg_borrow_archive
Immutable Archive

AFTER UPDATE OF "returnDate" ON "borrow" WHEN (returnDate was NULL)

Fires only when returnDate changes from NULL to a real date — a genuine return event. The WHEN clause prevents accidental fires. Collects member name and resource title, then inserts a permanent denormalised snapshot into borrowHistory.

The denormalisation is intentional: the archive record remains readable even if the original member or resource row is later deleted.

BEFORE UPDATE ON "resource"

A BEFORE trigger that raises ORA-20010 if any UPDATE would set availableCopies below zero. This is a belt-and-suspenders safety net: the borrowResource stored procedure already checks availability, and the CHECK constraint catches negatives — but this trigger catches any direct UPDATE that bypasses both.

Some might call three layers of protection on one column excessive. Those people have never seen a developer run a bulk UPDATE script at 11pm on a Friday without checking the WHERE clause first. We have seen this. Three layers is appropriate.
Section 12

Roles & Least-Privilege Access

Three layered database roles — because not everyone needs to be a database demigod

The Principle of Least Privilege

Every component should have access to exactly what it needs — no more. In database terms: create distinct roles for distinct user types and grant only the specific tables, columns, and procedures each role requires. If someone doesn't need DELETE, they don't get DELETE.

We have three roles: agri_readonly (can look, cannot touch), agri_member_role (can borrow and pay, cannot delete anything), and agri_admin_role (can do almost everything, but still cannot UPDATE or DELETE the blockchain table — because even Oracle wouldn't let us grant that, because it doesn't exist as a permission for that table type). We briefly considered an agri_omnipotent_god_role. We decided that was a naming convention we didn't want to defend in a code review.

Role Inheritance

Roles cascade: agri_admin_role inherits everything from agri_member_role, which inherits everything from agri_readonly. This means each level only defines its additions, and revoking a middle role cascades through the hierarchy.

Column-level grant highlight: agri_member_role receives UPDATE ("returnDate") ON "borrow" — not full UPDATE privilege on the table. Only that one column. Even if a member gains direct SQL access, they can only set a return date. They cannot change who borrowed the resource, what was borrowed, or when.
What not even agri_admin_role can do: UPDATE or DELETE rows in paymentAuditLedger. Not because we didn't grant it — because the blockchain table doesn't permit it as a concept. You cannot grant a permission that doesn't exist. The entire point.

Section 13 — Oracle 21c Feature

Property Graph

Relational data modelled as a graph — members and resources as vertices, borrows as edges

What is a Property Graph?

Oracle Property Graphs define a graph schema over existing relational tables. No data is copied. No ETL. The graph is a live view — vertices (nodes) map to tables, edges map to association tables, and the entire thing is queryable with SQL/PGQ pattern-matching syntax built directly into a regular SELECT statement.

SQL/PGQ lets you express things like: "find all members connected to this resource through borrow events" using graph pattern notation (m IS Member)-[b IS Borrowed]->(r IS Resource) — a style borrowed from Cypher (Neo4j's query language) but executed entirely inside Oracle.

Our graph model: Members, Resources, and Branches are vertex tables. The borrow table becomes an edge labelled "Borrowed", connecting Members to Resources. This enables graph-native queries: which resources are most central to the network? Which members share reading interests? What is the borrow-distance between two members?
Flask → D3 connection: The /api/graph-data endpoint runs three SQL queries — member nodes, resource nodes, borrow edges — and returns {"nodes": [...], "links": [...]}. D3's forceSimulation uses this to render an interactive, physics-based network graph on the admin dashboard. Drag a node. Watch the network reorganise. Oracle data. Browser physics. Magic.
Did we absolutely need a property graph for a library management system? Statistically: no. Could a GROUP BY have produced the same "top borrowers" list? Yes, obviously. Does a GROUP BY produce an interactive force-directed D3 network diagram on the admin dashboard where you can drag individual member nodes around and watch the graph physics respond in real time? It does not. The property graph is completely justified on aesthetic grounds alone. We stand by this decision.
Property Graph Definition + GRAPH_TABLE Query 
-- Oracle 21c: Property Graph over existing tables
CREATE OR REPLACE PROPERTY GRAPH "agriGraph"
    VERTEX TABLES (
        "member"
            KEY ("memberID")
            LABEL Member
            PROPERTIES ("firstName", "joinDate"),

        "resource"
            KEY ("resourceID")
            LABEL Resource
            PROPERTIES ("title", "type"),

        "branch"
            KEY ("branchID")
            LABEL Branch
            PROPERTIES ("branchName", "location")
    )
    EDGE TABLES (
        "borrow"
            KEY ("borrowID")
            SOURCE KEY      ("memberID")
                REFERENCES "member"
            DESTINATION KEY ("resourceID")
                REFERENCES "resource"
            LABEL Borrowed
            PROPERTIES ("borrowDate", "returnDate")
    );

-- Graph query using SQL/PGQ MATCH syntax
SELECT member_id, first_name, borrow_count
FROM GRAPH_TABLE (
    "agriGraph"
    MATCH
        (m IS Member)
        -[b IS Borrowed]->
        (r IS Resource)
    COLUMNS (
        m."memberID"  AS member_id,
        m."firstName" AS first_name,
        r."title"     AS resource_title,
        COUNT(*)     AS borrow_count
    )
)
ORDER BY borrow_count DESC
FETCH FIRST 10 ROWS ONLY;
-- The pattern (m)-[b]->(r) reads naturally:
-- "a Member connected via a Borrowed edge to a Resource"

D3.js Network Graph

The member–resource borrow network uses D3's forceSimulation — a physics engine where nodes repel each other and edges act as springs. The layout self-organises based on connection density. Highly connected resources drift to the centre; isolated nodes float to the edges. It is both analytically useful and genuinely beautiful to watch.

Data comes from /api/graph-data which queries member nodes, resource nodes, and borrow edges. Node colour encodes branch membership. Resource nodes use a 1000+ ID offset to avoid collision with member IDs across the two tables.

Drag interaction: D3's drag event updates node fx and fy (fixed position) in real time. Release the node and it rejoins the simulation. The graph is fully interactive — not a static image. Oracle data. Live physics. One fetch() call.
The membership growth chart has two datasets on the same axes: monthly new members (a line chart) and cumulative total members (a running sum). The cumulative total uses Oracle's SUM() OVER (ORDER BY month ROWS UNBOUNDED PRECEDING) window function. Window functions are one of those features that, once understood, make you wonder how databases existed without them. They're like GROUP BY but they don't discard your rows — they annotate them. Oracle has had window functions since version 8i (1997). They are 28 years old. We are using them in the freshest Oracle edition. They have aged like fine agricultural produce.
Section 15

How Everything Connects

Three complete end-to-end walkthroughs of the system's most important flows

Flow 1 — A Farmer Joins the Society

Step 1
Browser → GET /join

Flask calls getCategories() and getBranches() via SYS_REFCURSOR stored procedures. Results returned as list[dict], passed to join.html via Jinja2 context. The form renders with real membership categories and real branches from Oracle.

Step 2
Farmer fills form → POST /members/register

Flask calls addMember() stored procedure with form data. The procedure checks for a duplicate email — RAISE_APPLICATION_ERROR(-20001, 'Email already registered') if found — then INSERTs the new member with joinDate = SYSDATE. bcrypt hashes the password before the INSERT.

Step 3
INSERT fires trg_member_audit automatically

The AFTER INSERT trigger writes an 'INSERT' row to memberAuditLog with the database user, timestamp, and the new email value. Zero application code involved. The audit record exists before Flask has even returned its HTTP response.

Step 4
Flash message: "Welcome — your Member ID is 42"

Flask queries the newly created memberID by email, stores it in a flash message, and redirects to the login page. The farmer now has an ID. It is permanently recorded. The Oracle audit table already has proof it was created. Everyone is happy.

Step 5
Farmer logs in → POST /members/login

Flask queries member by UPPER("email") (hits the function-based index), retrieves "pwdHash" by explicit name (invisible column — not in SELECT *). bcrypt.checkpw() verifies. Flask session set. Member is in.

Step 6
Dashboard → GET /members/dashboard

Flask calls getMemberActivityReport() — this procedure uses the virtual fullName column and the member_tenure_years() SQL Macro. The member sees their profile, active borrows (from active_borrows_for() table macro), and payment history. Six Oracle 21c features, one dashboard.

Flow 2 — Member Makes a Payment

Step 1
POST /members/pay

Flask calls makePayment(memberID, amount, paymentType) — a stored procedure that validates the payment type, verifies the member is active, and INSERTs into the payment table. One stored procedure call. One INSERT.

Step 2
AFTER INSERT fires trg_payment_blockchain — automatically

Oracle immediately inserts a row into paymentAuditLedger — the blockchain table — computing and storing the SHA2_512 hash chain link. The Flask application cannot observe this happening. It cannot prevent it. It cannot intercept it. It simply occurs.

Step 3
Blockchain row is permanent

The paymentAuditLedger row cannot be deleted for 31 days, cannot be updated ever, and the table cannot be dropped while it contains recent data. Call DBMS_BLOCKCHAIN_TABLE.VERIFY_ROWS at any point in the future — Oracle will confirm the chain is intact.

Step 4
Admin views the ledger → GET /admin/audit

Flask calls getPaymentLedger() which SELECTs from the blockchain table. Admin sees the full immutable payment history: amount, type, member, timestamp, and the database user who triggered the write. This is the financial record of the society. It cannot be edited. Not even by the admin. Especially not by the admin.

Flow 3 — Admin Views the Dashboard

Step 1
GET /admin/ → admin_dashboard.html

Flask calls getDashboardStats(), getBranchReport(), getTopBorrowers(), and getOverdueBorrows() — all returning SYS_REFCURSOR results. Data is passed to the template as Python dicts. Jinja2 renders KPI cards with data-target attributes for JavaScript counter animations.

Step 2
JavaScript loads and initialises charts

charts.js runs after DOM load. Six chart initialisation functions are called, each triggering an AJAX fetch() to the corresponding /api/* endpoint. Flask calls the Oracle stored procedure or view for each, returns JSON via jsonify(). Chart.js renders on <canvas> elements.

Step 3
D3 network graph loads from /api/graph-data

initNetworkGraph() calls /api/graph-data. Flask runs three SQL queries (member nodes, resource nodes, borrow edges) and returns a combined JSON object. D3 builds a force simulation with forceLink, forceManyBody, and forceCenter, and renders an interactive SVG network onto the page. The admin can drag nodes, inspect connections, and see the social topology of the library.

Step 4
Admin navigates to Audit page → GET /admin/audit

Both the blockchain ledger (paymentAuditLedger) and the member change log (memberAuditLog) are displayed side by side. The blockchain ledger shows the hash-chained payment history. The audit log shows every member change with old and new values. Together they provide complete, tamper-evident accountability for all money and all member data in the system.

The Bottom Line

What We Actually Built

AgriSoc Kenya is a fully functioning agricultural society management system built on Oracle 21c and Flask. It uses blockchain tables to make payment history permanently tamper-proof, immutable tables to archive borrow history forever, native JSON columns for flexible announcement metadata stored in binary OSON, SQL macros for zero-overhead parameterised query logic inlined at parse time, virtual columns so full name is always in sync with no application effort, invisible columns so password hashes are never accidentally returned by a SELECT *, property graphs to model the borrow network and power the D3 visualisation, four triggers to automate everything the application shouldn't have to think about, nine indexes tuned to real query patterns including function-based and JSON path indexes, and three database roles giving each user type exactly the permissions they need — no more, no less.

The farmers of Kenya may not know their annual subscription fees are being preserved in a cryptographic chain. They may not know their membership tenure is calculated by a function that dissolves into the query plan at parse time and leaves no runtime trace. They may not know their borrow records are stored in an append-only table that will outlast the server it runs on.

They deserve nothing less.
